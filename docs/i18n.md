## ADR-015 — Biblioteca de i18n: react-i18next
**Data:** 2026-06-29 · **Status:** Aceito

**Contexto:** o frontend tem ~36 arquivos `.tsx` com texto **pt-BR hardcoded**. O desafio
pede a app em 3 idiomas. Não há lib de i18n no projeto hoje.

**Alternativas avaliadas (pesquisa 2026):**
- **react-i18next** (~22 kB) — maior ecossistema, padrão de mercado, JSON por idioma.
- **Lingui** (~10 kB) — menor bundle, mensagens co-localizadas, extração automática.
- **react-intl / FormatJS** (~22 kB) — ICU estrito, forte em formatação de data/número.

**Decisão:** **`react-i18next`**.

**Motivos:**
- Padrão de mercado — alinha com a lógica do **ADR-001** ("mais material, empregável").
- JSON por idioma combina com a convenção do projeto *identificador em inglês / texto em pt-BR*
  (a **chave** fica em inglês; o **valor** é o texto traduzido).
- Detecção de idioma + persistência em `localStorage` prontas (`i18next-browser-languagedetector`).
- Pluga direto no `SettingsPanel`, que **já tem a UX** de Select/RadioGroup.

**Trade-off aceito:** bundle maior que o Lingui (~22 vs ~10 kB). Aceitável; mitigado com
carregamento lazy por namespace. Reavaliar Lingui só se o bundle do PWA doer.

## ADR-016 — Estrutura e convenções de tradução
**Data:** 2026-06-29 · **Status:** Aceito

**Decisão:** locales em JSON, namespaces por feature, espelhando o padrão modular do projeto.

```
src/i18n/
├── index.ts            ← init do i18next (detector + fallback pt-BR + tipos)
└── locales/
    ├── pt-BR/  common.json · settings.json · query.json · notifications.json · map.json
    ├── en/     (mesmas chaves)
    └── es/     (mesmas chaves)
```

**Convenções:**
- **Chave em inglês, valor traduzido** — respeita a regra *identificador = inglês · texto que o
  usuário lê = pt-BR/en/es*. Ex.: `t("settings:monitoredRegion")`.
- **Namespace por feature** (`settings`, `query`, `notifications`, `map`, `common`) — combina com `features/`.
- **Carregamento lazy por namespace** — não infla o chunk inicial (alinha com o code-splitting do `router.tsx`).
- **Chaves tipadas** (`resources` tipados no `i18next`) → o `tsc -b` pega chave inexistente
  (cultura de "validar com build" do projeto).
- **Wiring:** `import "./i18n"` no `src/main.tsx` (ao lado do `@fontsource`).
- **Detecção:** navegador → fallback **pt-BR**; persistência em `localStorage` (`appbit.lang`).

## ADR-017 — Mocks e conteúdo dinâmico
**Data:** 2026-06-29 · **Status:** Aceito

**Decisão:** separar **casca** (i18n cobre) de **conteúdo dinâmico** (vem do backend).

- **Mocks** (`notifications/mock.ts`, sugestões do `query-flow`): trocar strings fixas por
  **chaves de tradução**, não texto cru. Quando o backend real entrar, ele devolve dados já
  localizados ou um código que vira chave.
- **Conteúdo da IA** (`QueryResult` / `postQuery`): o texto vem do backend → i18n cobre só a
  **casca** (rótulos, botões, status). Decidir depois se o backend recebe `Accept-Language`.
- **PWA:** os JSON são puxados via fetch; para funcionar offline, **bundlar os 3 idiomas** ou
  **pré-cachear** os JSON no service worker (quando o `vite-plugin-pwa` entrar — ADR-009).

## Plano de implementação (ondas)

- **Onda 0 — 1ª entrega (PRIORIDADE): ✅ FEITA (2026-06-29).** `react-i18next` +
  `i18next-browser-languagedetector` instalados; setup em `src/i18n/` (`index.ts` + `i18next.d.ts`
  com chaves tipadas); locales `common`/`settings` nos 3 idiomas; `import "./i18n"` no `main.tsx`;
  **seletor de idioma no `SettingsPanel`** (Select 🇧🇷/🇺🇸/🇪🇸 → `i18n.changeLanguage`, persiste em
  `localStorage` `appbit.lang`). `SettingsPanel` 100% traduzido como feature de exemplo (valores de
  Select agora são chaves estáveis em inglês, rótulo via `t`). Validado com `npm run build` + `lint`.
  - **Nota de implementação:** resources **bundlados** (import estático dos JSON) em vez de http-backend
    lazy — mais simples e já funciona offline. `react.useSuspense: false`. O lazy por namespace
    (ADR-016) fica como otimização futura quando o volume de strings crescer.
- **Onda 1 — casca: ✅ FEITA (2026-06-29).** Namespaces `nav` (sidebar + ações topbar) e `signIn`
  (3 idiomas) + `common.underConstruction`. `AppLayout` traduz `NAV_ITEMS` (labelKey → `t`, reativo
  ao idioma; alimenta sidebar + título da topbar) e os `IconButton` de config/sino; `SignInPage`
  100% traduzida (hero, quote, form, gov.br, rodapé; saudação via interpolação `{{name}}`);
  `PlaceholderPage` recebe `titleKey: keyof typeof nav` (chave tipada) e usa `common:underConstruction`;
  `router.tsx` passa `titleKey` ("analytics"/"reports"/"alerts"). **`TopAppBar` NÃO foi tocado** —
  é primitivo do DS (recebe texto por prop), traduzido no call site (regra "layout não define primitivo").
  Validado com build + lint.
- **Onda 2 — features: ✅ FEITA (2026-06-30).** Namespaces `query` (4 steps) e `notifications`
  (chrome + abas + itens mockados) nos 3 idiomas; `common` +`askAI`/`send`/`promptPlaceholder`.
  `query-flow`: InputStep/LoadingStep/ResultStep/ExportStep traduzidos (sugestões e fases via
  `returnObjects`; `result.meta` com interpolação `{{time}}`/`{{count}}`; PDF imprimível também).
  `notifications`: `types.ts`/`mock.ts` viraram **só metadados** (id/severity/category/hasAction/
  unread); o texto saiu do mock e mora no i18n em `items.<id>.*`, resolvido no painel por `id`
  (chave dinâmica tipada porque `NotificationId` é union). `AIPrompt` (primitivo do DS) traduz
  placeholder + a11y via `common`. **MapPage** não tem texto próprio (tooltip = nome de bairro =
  dado) → sem ns `map`. Validado com build + lint.
  - **Fora de escopo (não renderizados):** `AlertBadge`, `AIQueryBar`, `MapPin`, `KpiCard` existem
    na lib do DS mas não são montados em nenhuma página → traduzir só quando forem usados.
  - **Mantido cru de propósito:** `api/endpoints.ts` `postQuery` (mock) devolve `QueryResult` em
    pt-BR = **conteúdo dinâmico da IA → responsabilidade do backend** (ADR-017). Em sessão en/es a
    casca do resultado traduz, mas o `claim`/evidência do mock seguem pt-BR até o backend existir.
- **Tradução en/es:** pode entrar junto ou depois (placeholders primeiro; pt-BR sempre completo).

## Pegadinhas previstas

- **Build estrito:** tipar `resources` para o `tsc -b` validar as chaves (mais rígido que o dev).
- **PWA/offline:** JSON via fetch não fica offline sozinho — bundlar ou pré-cachear.
- **Fallback sempre pt-BR:** garante que nada apareça "cru" se faltar chave em en/es.
- **Não traduzir dados da IA no front** — isso é responsabilidade do backend.
